> 챕터 본문(`src/content/chapters/*.ts`)을 쓸 때 지키는 규칙.

> 여기 있는 항목은 **전부 실제로 한 번씩 어긴 것**이다. 추상적인 이상론이 아니라 재발 방지 목록이다.

원본 교재 집필 원칙은 `TEACHING_STYLE.md`(외부 문서)에 있고, 이 문서는 **그걸 앱 화면에 옮길 때**의 규칙이다.

## 0. 대전제

**읽는 사람은 일본어를 모른다.** N5 챕터를 읽는 사람은 히라가나도 갓 뗐다.
"이 정도는 알겠지"가 성립하는 지점이 **없다.**

**되돌아가게 만들지 않는다.** 위로 스크롤해서 확인해야 이해되는 문장은 실패한 문장이다.
각 블록은 그 자리에서 완결돼야 한다.

## 1. 일본어를 화면에 놓는 법 ★가장 자주 어김

**일본어는 절대 맨몸으로 두지 않는다.** 예외 없다.

### 문장이면 → `ex` 블록

독음·뜻·재생 버튼이 자동으로 붙는다. 프로즈 안에 일본어 문장을 넣지 않는다.

```ts
// ❌ 이렇게 하지 말 것
{ kind: 'p', text: '「タイ料理は食べません」처럼 주어가 생략된 문장도 많아요.' }

// ✅ 이렇게
{ kind: 'p', text: '일본어엔 **주어가 아예 없고 주제만 있는** 문장도 많아요.' },
{
  kind: 'ex',
  jp: 'タイ料理は食べません',
  read: '타이료-리와 타베마셍',
  mean: '태국 요리**는** 안 먹어요',
  mark: 'は', markTone: 'blue',
},
{ kind: 'p', text: '"누가" 안 먹는지가 없어요. 「私が(와타시가, 내가)」가 통째로 빠졌는데도 말이 돼요.' },
```

**한 문단에 예문 여러 개를 뭉치지 않는다.** 셋이면 `ex` 셋으로 나눈다.
(실제 사고: 🌉 콜아웃에 「あ、手紙がある」「雨が降ってる」「雨は降ってる」가 한 줄에 뭉쳐 있었다.)

### 단어·조각이면 → 괄호 독음

프로즈 안에서 단어를 언급할 땐 **한글 독음을 괄호로** 붙인다. 뜻도 필요하면 같이.

```
降る(후루, 내리다) / 私(와타시, 나) / 「ご注文は何ですか(고츄-몬와 난데스카, 주문은 무엇입니까)」
```

### 표 안이면 → 세 줄로 쌓는다

`grid` 셀은 `\n`으로 줄바꿈된다. **일본어 / 독음 / 뜻** 순서로.

```ts
cells: ['お酒**を**飲む\n오사케오 노무\n술을 **마시다**', ...]
```

### 예외: 트리(`tree`)

모노스페이스 정렬이 깨지므로 독음을 붙이지 않는다.
단 **트리에 처음 나오는 일본어를 넣지 않는다** — 본문에서 이미 다룬 것의 요약으로만 쓴다.

## 2. `ex` 블록 쓰는 법

### 독음과 뜻은 반드시 분리

```ts
// ❌ 한 줄에 붙이면 눈이 어디를 봐야 할지 모른다
read: '무카시, 오지이산가 이타 · 옛날에 할아버지가 있었다'

// ✅
read: '무카시, 오지이산가 이타',
mean: '옛날에 할아버지**가** 있었다',
```

**시각 위계는 일본어 > 뜻 > 독음.** 독음은 보조 정보라 가장 흐리다.
독음이 뜻만큼 진하면 둘 다 안 읽힌다.

### 강조는 색까지 맞춘다

`mean`의 `**굵게**`는 `markTone` 색으로 칠해진다.
일본어의 「が」가 주황이면 뜻의 "**가**"도 주황 → 눈이 둘을 자동으로 잇는다.
**조사 대응을 보여주는 예문이면 양쪽 다 강조를 넣는다.**

| markTone | 쓰는 곳 |
|---|---|
| `warn` (주황) | が, を, 함정·주의 |
| `blue` (파랑) | は, 정상·기본형 |
| `good` (초록) | で |

### 틀린 예문은 보여주되, 귀에 심지 않는다

오답 유도(설계원칙 ①)를 위해 **틀린 문장을 보여주는 건 좋다.** 단:

- `mean`에 **❌**와 **왜 틀렸는지**를 적는다
- **`speak`는 맞는 문장으로 덮어쓴다** — 틀린 발음을 귀에 남기면 안 된다

```ts
{
  kind: 'ex',
  jp: '誰は来ましたか',
  read: '다레와 키마시타카',
  mean: '❌ 누구**는** 왔어요? — 일본어엔 **없는 문장**이에요',
  speak: '誰が来ましたか',   // ← 재생하면 맞는 쪽이 나온다
  mark: 'は', markTone: 'blue',
},
```

## 3. 탭별 역할 분담

세 탭은 **절대 통합하지 않는다** (TEACHING_STYLE 최상위 규칙).
각 탭에 뭘 넣을지 헷갈리면 이 기준으로 자른다.

| 탭 | 넣는 것 | 넣지 않는 것 |
|---|---|---|
| 🇰🇷 한국어 | **한국어 감이 배신하는 곳.** 함정, 예외, 대응 안 되는 지점 | 한국어와 똑같은 것 |
| 🇯🇵 일본식 | 국어문법 용어(主語·未然形…), 왜 그런지의 뿌리, 역사 | 한국어 비교 |
| 📌 정리 | 트리 + 핵심 3줄 + 다음 챕터 | 새 정보 |

**🇰🇷 탭에 "이건 한국어랑 똑같아요"가 들어가면 자리를 잘못 잡은 것이다.**

실제 사고: 象は鼻が長い가 🇰🇷 탭에 있었다. "사실 한국어랑 똑같다"는 내용이라
"배신하는 곳만 모은다"는 🇰🇷 탭 성격과 어긋났다 → 🇯🇵 탭 主題+主語 문형으로 옮김.
옮기니 🇰🇷 탭이 **예외 3곳으로 딱 떨어졌다.**

### 한 줄 요약(파란 박스) vs 정리 탭

역할이 다르므로 규칙도 다르다.

- **한 줄 요약**은 읽기 **전**에 본다 → "예외 3곳만 챙기면 돼요" 같은 **예고**로 OK.
  겁먹지 말라는 신호라, 여기서 이름까지 대면 김이 빠진다.
- **정리 탭**은 읽은 **후**에 본다 → 아래 규칙을 따른다.

## 4. 정리 탭 `keys` 쓰는 법 ★두 번 연속 걸린 곳

### 항목은 자기 완결적이어야 한다

되돌아봐야 아는 참조를 쓰지 않는다.

```ts
// ❌ "3곳"이 뭔지 위로 올라가야 안다
'**は≈는 / が≈가**로 대부분 맞아요. 밀도는 예외 3곳에만 쏟으면 돼요.'

// ✅ 이름을 댄다
'**は≈는 / が≈가**로 대부분 맞아요. 새로 외울 건 한국어 감이 배신하는 세 곳 — **의문사 · 好き류 · 대조** — 뿐이에요.'
```

```ts
// ❌ 번호로 가리키기
'... (예외②는 이 원리)'

// ✅ 풀어쓰기
'... 好き·わかる·가능형이 が인 이유가 이거예요.'
```

### 규칙엔 예시를 붙인다

`\n`으로 줄을 나눠 규칙 밑에 예시를 둔다. 예시 찾으러 위로 올라가지 않게.

```ts
'**새 정보가 조사보다 앞에 있으면 が, 뒤에 있으면 は.**\n「雨**が**降ってる」는 "비"가 새 정보 → が보다 **앞**\n「私**は**食べました」는 "먹었다"가 새 정보 → は보다 **뒤**'
```

### 기준을 빼먹지 않는다

"앞이면 が, 뒤면 は" — **뭐의** 앞뒤인지가 없으면 문장이 무의미해진다.
(원문엔 "**조사** 앞/뒤"였는데 요약하며 "조사"를 흘렸다.)

**요약할 때 명사 하나 빠지면 문장 전체가 죽는지 확인할 것.**

## 5. 쓰지 말 것

### 세어보지 않은 숫자

> ❌ "규칙 다섯 개가 이 한 감각으로 수렴해요"

실제로 세어보니 5개 중 3개만 수렴했다. 好き류는 스포트라이트가 아니라 **동작/상태** 원리다.

**"N개가 ~한다"고 쓰면 진짜 세어본다.** 안 맞으면 범위를 정확히 적는다.
멋있어 보이는 요약을 근거 없이 붙이지 않는다.

### 설계 용어 누출

> ❌ "**밀도**는 예외 3곳에만 쏟으면 돼요"

"학습 밀도"는 설계 원칙(모국어와 같은 건 빠르게·다른 건 깊게)의 **내부 용어**다.
학습자는 이 프레임을 모른다. **설계 노트의 말이 화면에 새어나오지 않게 한다.**

같은 부류: "원칙 ③ 짝 설계", "아이템 모델", "척추 필터" 등.
(단 **"척추"는 학습자용 개념으로 승격**시켜 쓰기로 한 용어라 예외.)

### 그 외

- 표만 던지고 넘어가기 → 표 위에 **외울 리듬**(`rhythm` 블록)이나 감각 한 줄을 먼저 놓는다
- 정중체 남용 → 반말 섞인 `~해요`체. 딱딱하면 안 읽힌다
- 공격적 어휘 → 활용표에 `死ね` 같은 건 넣지 않는다 (문법상 정칙이어도)

## 6. 코드 쪽 주의

### `**굵게**`는 `inline()`을 통과할 때만 강조다

`inline()`을 안 거치는 필드에 쓰면 **별표가 화면에 그대로 찍힌다.**
(실제 사고: `read` 필드가 raw 렌더링이라 `**가**`가 별표째 보이고 있었다.)

**새 텍스트 필드를 추가하면 렌더러에서 `inline()`을 연결했는지 반드시 확인한다.**

### 화면 상태는 챕터가 바뀌면 리셋한다

`/grammar/:id`는 라우트 패턴이 같아서 **챕터를 넘어가도 컴포넌트가 유지된다.**
탭·스크롤을 명시적으로 초기화하지 않으면 정리 탭에서 넘어갔을 때
다음 챕터도 정리 탭 중간부터 보인다. (`ChapterPage`의 `useEffect([id])`)

### 스키마를 늘릴 땐 선택 필드로

`mean`, `rhythm`, `grid`, `bridge`처럼 나중에 추가한 것들은 전부 optional이라
기존 챕터가 안 깨졌다. **하위호환을 유지하면 챕터를 하나씩 고쳐나갈 수 있다.**

## 7. 새 챕터 체크리스트

본문을 다 쓴 뒤 이 순서로 훑는다.

- [ ] 화면의 **모든 일본어**에 독음이 붙어 있나 (프로즈 안 포함)
- [ ] 예문에 뜻(`mean`)이 있나. 독음과 한 줄에 뭉쳐 있지 않나
- [ ] 조사 대응 예문인데 한쪽만 강조돼 있지 않나 (`mark` + `mean`의 `**`)
- [ ] 프로즈에 일본어 **문장**이 박혀 있지 않나 → `ex`로 승격
- [ ] 🇰🇷 탭에 "한국어와 똑같은 것"이 섞여 있지 않나 → 🇯🇵로
- [ ] 정리 탭 항목이 **그 자리에서** 이해되나 (숫자·번호 참조 없나)
- [ ] "N개가 수렴/해당" 썼으면 **진짜 세어봤나**
- [ ] 표 위에 리듬이나 감각 한 줄이 있나
- [ ] `next` 카드가 실재하는 챕터 id를 가리키나
- [ ] `npm run typecheck` 통과
- [ ] **실제 화면에서 한 번 읽어보기** — 위 항목은 코드만 봐선 안 걸린다
